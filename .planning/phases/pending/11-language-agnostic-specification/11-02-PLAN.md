    Extract algorithm logic from C# scorer source files and write specification chapters. Each chapter follows the dual-audience pattern from 11-RESEARCH.md.

    1. Write `spec/src/scorers.md` overview:
       - Define the scorer interface contract: `Score(item, allItems) -> float64`
       - Scorer receives the item to score AND the full list of scoreable items (enables relative scoring)
       - Output conventionally [0.0, 1.0], documented but not enforced by type
       - Scorers are pure functions (no side effects, no state mutation between calls)
       - Table summarizing all 8 scorers with inputs used and output range

    2. Write `spec/src/scorers/recency.md` (RecencyScorer):
       - Algorithm: rank-based. Rank = count of items with strictly older timestamp.
       - Score = `rank / (countWithTimestamp - 1)`
       - Null timestamps → 0.0
       - Single item with timestamp → 1.0
       - All null timestamps → all 0.0
       - CLRS-style pseudocode

    3. Write `spec/src/scorers/priority.md` (PriorityScorer):
       - Algorithm: rank-based. Rank = count of items with strictly lower priority value.
       - Score = `rank / (countWithPriority - 1)`
       - Null priorities → 0.0
       - Single item with priority → 1.0
       - Higher numeric priority = higher score
       - CLRS-style pseudocode

    4. Write `spec/src/scorers/kind.md` (KindScorer):
       - Algorithm: dictionary lookup. `weights[item.kind]` or 0.0 if kind not in weights.
       - Default weights: SystemPrompt=1.0, Memory=0.8, ToolOutput=0.6, Document=0.4, Message=0.2
       - ContextKind comparison is case-insensitive (P4)
       - Custom weights configurable at construction
       - CLRS-style pseudocode

    5. Write `spec/src/scorers/tag.md` (TagScorer):
       - Algorithm: sum matched tag weights, normalize by total configured weight, clamp to [0.0, 1.0]
       - `score = min(sum(matched_weights) / total_configured_weight, 1.0)`
       - Tag comparison: case-insensitive
       - No tags → 0.0
       - No configured weights → 0.0
       - CLRS-style pseudocode

    6. Write `spec/src/scorers/frequency.md` (FrequencyScorer):
       - Algorithm: proportion of peer items sharing at least one tag with this item
       - `score = matchingPeers / (allItems.count - 1)`
       - Tag comparison: case-insensitive
       - Self excluded from peer count (by identity, not by value)
       - No tags → 0.0; single item → 0.0
       - CLRS-style pseudocode

    7. Write `spec/src/scorers/reflexive.md` (ReflexiveScorer):
       - Algorithm: passthrough of `item.futureRelevanceHint`, clamped to [0.0, 1.0]
       - Null or non-finite (NaN, infinity) → 0.0
       - CLRS-style pseudocode

    8. Write `spec/src/scorers/composite.md` (CompositeScorer):
       - Algorithm: weighted average of child scorer outputs
       - `score = sum(child_i.Score(item, allItems) * normalizedWeight_i)`
       - Weights normalized to sum to 1.0 at construction time (relative, not absolute)
       - Nested composites supported (composite containing composites)
       - DAG cycle detection at construction time (DFS traversal)
       - CLRS-style pseudocode for both construction validation and scoring

    9. Write `spec/src/scorers/scaled.md` (ScaledScorer):
       - Algorithm: min-max normalization of an inner scorer across all items
       - First pass: score all items with inner scorer to find min and max
       - Second pass: `(raw - min) / (max - min)` for each item
       - Degenerate case (all scores equal, max == min): return 0.5
       - O(N) per item, O(N^2) total
       - CLRS-style pseudocode

    1. Write `spec/src/slicers.md` overview:
       - Define the slicer interface contract: `Slice(scoredItems, targetTokens) -> selectedItems`
       - Slicer receives pre-sorted scored items and a token budget
       - Must return a subset that fits within targetTokens
       - Table summarizing all slicers with complexity and use case
       - Note: StreamSlice is documented for completeness but marked as implementation-optional (async/streaming path is out of spec scope for synchronous pipeline)

    2. Write `spec/src/slicers/greedy.md` (GreedySlice):
       - Algorithm: sort by value density (score/tokens), greedily fill
       - Value density = `score / tokens`; zero-token items = MAX_FLOAT density (always included)
       - Stable sort descending by density, tiebreak by ascending index
       - Iterate: include item if `tokens <= remaining`; zero-token items always included
       - O(N log N) complexity
       - CLRS-style pseudocode (use the example from 11-RESEARCH.md as starting point)

    3. Write `spec/src/slicers/knapsack.md` (KnapsackSlice):
       - Algorithm: 0/1 knapsack dynamic programming
       - Score scaling: multiply by 10000 and truncate to integer for DP table
       - Weight discretization: item weights ceiling-divided by bucketSize; capacity floor-divided by bucketSize
       - This guarantees feasibility: selected items always fit within original budget
       - Zero-token items pre-filtered (always included, not part of DP)
       - 1D DP array with reverse iteration; 2D boolean keep table for reconstruction
       - Address P5: spec defines behavioral contract (optimal or near-optimal within budget) but notes that internal precision (score scaling factor, bucket size) is implementation-defined. Conformance tests use scores separated by >= 0.01
       - CLRS-style pseudocode

    4. Write `spec/src/slicers/quota.md` (QuotaSlice):
       - Algorithm: decorator pattern. Partitions items by ContextKind, computes per-kind token budgets, delegates per-kind to inner slicer
       - Require(kind, minPercent): guarantees minimum percentage of budget for a kind
       - Cap(kind, maxPercent): limits maximum percentage of budget for a kind
       - Budget distribution:
         a. Allocate Require minimums first (floor truncation)
         b. Remaining budget distributed proportionally by candidate token mass per kind
         c. Cap constraints applied as ceiling
       - Address P6: floor truncation means sum of kind budgets may be less than targetTokens
       - Address P4: ContextKind comparison is case-insensitive throughout
       - Validation: sum of Require percentages must not exceed 100%
       - Pinned items that conflict with quota constraints: produce diagnostic warning, do not silently fail
       - CLRS-style pseudocode

    5. Write `spec/src/placers.md` overview:
       - Define the placer interface contract: `Place(items) -> orderedItems`
       - Placer determines final ordering of selected items in the context window
       - Does not add or remove items — only reorders
       - Table summarizing both placers

    6. Write `spec/src/placers/chronological.md` (ChronologicalPlacer):
       - Algorithm: stable sort ascending by timestamp
       - Null timestamps sort to end (after all timestamped items)
       - Tiebreak: ascending by original index
       - CLRS-style pseudocode

    7. Write `spec/src/placers/u-shaped.md` (UShapedPlacer):
       - Algorithm: exploits primacy + recency attention bias
       - Stable sort descending by score
       - Even-indexed items (0, 2, 4, ...) placed at left edge (start of context)
       - Odd-indexed items (1, 3, 5, ...) placed at right edge (end of context), in reverse order
       - Result: highest scores at both edges, lowest scores in the middle
       - Include a diagram or ASCII art showing the U-shaped placement pattern
       - CLRS-style pseudocode
