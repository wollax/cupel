    TotalTokens computation:
    - TotalTokens returns sum of all Items' Tokens values
    - TotalTokens with empty Items returns 0
    - TotalTokens with single item returns that item's Tokens
    - TotalTokens with multiple items returns correct sum

    Immutability:
    - ContextResult supports with-expressions: `result with { Report = newReport }`

    SelectionReport:
    - Construction with required Events list
    - Events list is accessible after construction

    **IMPORTANT — TotalTokens implementation:**
    Do NOT use LINQ `Sum()` — it allocates a delegate. Use a manual for loop or compute-and-cache at init time. See 02-RESEARCH.md Pitfall 2.

    Recommended approach — compute once via init accessor with `field` keyword (C# 13 / .NET 10):
    ```csharp
    public sealed record ContextResult
    {
        private readonly int _totalTokens;

        public required IReadOnlyList<ContextItem> Items
        {
            get;
            init
            {
                field = value;
                _totalTokens = ComputeTotal(value);
            }
        }

        public int TotalTokens => _totalTokens;

        public SelectionReport? Report { get; init; }

        private static int ComputeTotal(IReadOnlyList<ContextItem> items)
        {
            var total = 0;
            for (var i = 0; i < items.Count; i++)
                total += items[i].Tokens;
            return total;
        }
    }
    ```

    If the `field` keyword approach causes issues with the record's generated `Equals`/`PrintMembers`, fall back to a computed property with a manual loop (no caching):
    ```csharp
    public int TotalTokens
    {
        get
        {
            var total = 0;
            for (var i = 0; i < Items.Count; i++)
                total += Items[i].Tokens;
            return total;
        }
    }
    ```

    The loop approach is acceptable because ContextResult is typically accessed once for TotalTokens. Either approach satisfies the no-LINQ constraint.
