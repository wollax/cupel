2. Refactor `AddCupelPipeline` in `CupelServiceCollectionExtensions.cs`:

   Create an internal record or class to hold the built pipeline components:
   ```csharp
   internal sealed record PolicyComponents(
       IScorer Scorer,
       ISlicer Slicer,
       IPlacer Placer,
       IAsyncSlicer? AsyncSlicer,
       bool DeduplicationEnabled,
       OverflowStrategy OverflowStrategy);
   ```
   (Place this as a nested private class or a separate internal class in the DI project.)

   Change the registration pattern:
   - Register a keyed **singleton** `PolicyComponents` that builds via `PipelineBuilder.WithPolicy().Build()`, extracts the components from the built pipeline (using internal access), and caches them.
   - Register a keyed **transient** `CupelPipeline` that resolves the singleton `PolicyComponents` and creates a new `CupelPipeline` instance using the internal constructor with those shared components + the budget.

   The approach relies on `CupelPipeline`'s internal constructor being accessible via `InternalsVisibleTo`. The internal constructor signature is:
   ```csharp
   internal CupelPipeline(
       IScorer scorer, ISlicer slicer, IPlacer placer,
       ContextBudget budget, bool deduplicationEnabled,
       IAsyncSlicer? asyncSlicer, OverflowStrategy overflowStrategy,
       Action<OverflowEvent>? overflowObserver)
   ```

   The singleton components approach: build a pipeline once to extract components, then create new pipeline instances reusing those same component references. Since `IScorer`, `ISlicer`, `IPlacer` implementations are stateless and thread-safe, singleton sharing is safe.

   Alternative simpler approach if the above is too complex: Use `Lazy<CupelPipeline>` in the factory closure to build once, then on each resolve, create a new `CupelPipeline` using the internal constructor with the lazy value's components. The `Lazy<T>` ensures thread-safe one-time initialization.

   **Key constraint:** The `Action<OverflowEvent>? overflowObserver` parameter — policies don't carry overflow observers (those come from builder API). Pass `null` for the observer in the DI path. This matches the current behavior since `WithPolicy()` calls `WithOverflowStrategy(strategy)` without an observer.

3. Pipeline must remain transient — each `GetRequiredKeyedService<CupelPipeline>(intent)` returns a **new** instance. Only the inner components (scorer, slicer, placer) are shared singletons.

4. `AddCupelTracing` with `TryAddTransient<ITraceCollector>` stays unchanged — ITraceCollector remains transient.
