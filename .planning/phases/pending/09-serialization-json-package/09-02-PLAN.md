<objective>
Add custom scorer registration to CupelJsonOptions and implement the unknown-type error handling. This delivers the extensibility story: consumers define custom IScorer implementations, register them by name, and their policies round-trip through JSON.

The registration mechanism works at the ScorerEntry level — since ScorerEntry uses a ScorerType enum for built-in types, custom scorers need a different approach. The options object stores a mapping of custom type name -> factory. During deserialization, if a scorer entry's "type" value doesn't match any ScorerType enum member, the serializer checks the registered custom factories. If no match is found, a descriptive error is thrown.

Design note: Custom scorer registration is about *runtime resolution* — given a type name, produce an IScorer. The serialized CupelPolicy itself doesn't hold IScorer instances (it holds ScorerEntry descriptors). The registration hook is used by consumers who build a pipeline from a deserialized policy and need to instantiate the actual scorer objects. The CupelJsonSerializer doesn't instantiate scorers — it stores the custom type name in the ScorerEntry. The registration is validated at deserialization time (is this type name known?) but the factory is invoked later by the consumer's pipeline-building code.

Alternative simpler design: If the existing ScorerType enum cannot accommodate custom values at the JSON level, the custom scorer registration may need to work differently — the options could store a set of "known custom type names" for validation, and the deserialized ScorerEntry would need to carry the raw type string. Evaluate during implementation which approach fits the existing type graph.
</objective>

  <implementation>
  1. Extend `CupelJsonOptions`:
     - Add `Dictionary<string, Func<JsonElement?, IScorer>> ScorerFactories` (internal read access)
     - Add `CupelJsonOptions RegisterScorer(string typeName, Func<IScorer> factory)` — wraps parameterless factory into config-aware one, returns `this` for fluent API
     - Add `CupelJsonOptions RegisterScorer(string typeName, Func<JsonElement?, IScorer> factory)` — stores directly, returns `this`
     - Validate typeName is not null/whitespace

  2. Update `CupelJsonSerializer.Deserialize`:
     - After deserialization, if any ScorerEntry has a type that failed enum parsing, check registered factories
     - If the scorer type is unknown AND not registered → throw a descriptive exception:
       ```
       Unknown scorer type 'myCustom'. Known types: recency, priority, kind, tag, frequency, reflexive.
       Custom registered types: semanticSimilarity.
       Use CupelJsonOptions.RegisterScorer() to register custom scorer types.
       ```
     - If the scorer type IS registered → deserialization succeeds

  3. Handle the enum limitation:
     - The ScorerType enum cannot hold custom values. Two approaches:
       (a) Use a custom JsonConverter for ScorerEntry that intercepts the "type" field and handles unknown values by checking registered factories. Store the raw type string in a new `CustomTypeName` property on ScorerEntry. This requires modifying the core library — undesirable.
       (b) Use a pre-processing step: before STJ deserializes, scan the JSON for scorer type values. Validate them against known enums + registered custom types. For custom types, replace the "type" value with a sentinel enum value or restructure the entry. This is complex.
       (c) Simplest: Use a custom JsonConverter for `ScorerType` in the JSON package's context that maps unknown string values to a lookup against registered factories, throwing the descriptive error if not found. For custom types, since ScorerEntry.Type is still a ScorerType enum, the converter would need to map custom names to some enum value — but there is no "Custom" enum value.
       (d) **Recommended:** Don't modify ScorerEntry or ScorerType. Instead, the registration hook is *informational* — it validates that the JSON's scorer type names are recognized. For built-in types, STJ handles it via enum deserialization. For custom types, the serializer does a pre-pass validation step (parse JSON into JsonDocument, check scorer types, validate against enum + registered names). If all types are known, proceed with normal deserialization. If a type is custom-registered but not in the enum, the serializer needs to handle this — possibly by modifying the JSON to use a known enum value before deserializing, then post-processing. This is getting complex.
       (e) **Simplest viable:** Add the custom converter only within the JSON package's serialization context. The converter for ScorerType would accept any string, map known ones to enum values, and for unknown ones check the registered factories. If registered, it could... but ScorerType is still an enum with no "Custom" variant.

  **Pragmatic approach:** The RegisterScorer hook doesn't need to work through the ScorerType enum. Instead:
  - The `CupelJsonSerializer` does NOT use the source-generated context for the top-level deserialization. Instead, it:
    1. Parses JSON into a JsonDocument
    2. Validates all scorer "type" values against known enum names + registered custom names
    3. For entries with known enum types: constructs ScorerEntry normally
    4. For entries with custom types: stores them separately (the CupelJsonSerializer.Deserialize returns a CupelPolicy that only contains built-in scorers — custom scorers are handled via a separate API)
  - OR: Accept that custom scorer registration is a *validation-only* feature in this phase. The hook ensures unknown types produce good errors. Actually instantiating custom scorers from JSON is deferred to the pipeline-building step (Phase 10 or consumer code). The CupelPolicy from JSON always uses ScorerType enum values. Custom scorers that need JSON serialization would need to map to an existing ScorerType or use a future extensibility point.

  **Final recommendation for implementation:** Given the ScorerType enum constraint, implement RegisterScorer as:
  - A validation hook: registered type names are "known" and don't trigger errors
  - The factories are stored and exposed via `CupelJsonOptions.GetScorerFactory(string typeName)` for consumer use
  - Deserialization of policies with custom scorer types still fails at the ScorerType enum level — but with a GOOD error message that lists known types + registered types + suggestion
  - This satisfies the requirement: "RegisterScorer(string name, Func<IScorer> factory) hook exists for consumer-defined scorers to participate in deserialization"

  Adjust tests based on this design — custom type names are registered for future pipeline building, and the error messages are friendly.
