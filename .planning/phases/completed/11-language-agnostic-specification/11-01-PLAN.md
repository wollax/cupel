    3. Write `spec/src/introduction.md`:
       - Declare spec version 1.0.0
       - State purpose: language-agnostic specification for the Cupel context selection algorithm
       - Define scope: synchronous pipeline, all scorers/slicers/placers, data model, overflow handling
       - State conformance model: behavioral equivalence (same items selected in same order), not bit-exact scores
       - Mandate IEEE 754 64-bit doubles for all scoring computations
       - State notation conventions: CLRS-style pseudocode, 0-based indexing, `<-` for assignment
       - Include a Mermaid flowchart showing the 6-stage pipeline (Classify → Score → Deduplicate → Sort → Slice → Place)

    5. Write `spec/src/data-model/context-item.md`:
       - Define all ContextItem fields with types, nullability, and constraints (extract from C# source)
       - `content`: string, non-nullable, non-empty
       - `tokens`: integer, non-nullable, required (caller-provided token count)
       - `kind`: ContextKind (extensible string, case-insensitive)
       - `source`: ContextSource (extensible string, case-insensitive)
       - `priority`: optional integer (higher = more important)
       - `tags`: list of strings (may be empty, case-insensitive comparison)
       - `metadata`: map of string to any (opaque, not used by pipeline)
       - `timestamp`: optional datetime (UTC)
       - `futureRelevanceHint`: optional float64 (conventionally 0.0-1.0)
       - `pinned`: boolean (default false)
       - `originalTokens`: optional integer (pre-compression token count)
    6. Write `spec/src/data-model/context-budget.md`:
       - `maxTokens`: integer >= 0 (hard ceiling)
       - `targetTokens`: integer >= 0, <= maxTokens (soft target for slicer)
       - `outputReserve`: integer >= 0, <= maxTokens (tokens reserved for model output)
       - `reservedSlots`: map of ContextKind to integer (minimum guaranteed items per kind)
       - `estimationSafetyMarginPercent`: float64 0-100 (buffer for token estimation error)
       - Document validation rules (targetTokens <= maxTokens, outputReserve <= maxTokens)
    7. Write `spec/src/data-model/enumerations.md`:
       - ContextKind: extensible string with well-known values (Message, Document, ToolOutput, Memory, SystemPrompt). Case-insensitive ASCII comparison.
       - ContextSource: extensible string with well-known values (Chat, Tool, Rag). Case-insensitive ASCII comparison.
       - OverflowStrategy: closed enum (Throw, Truncate, Proceed). Define behavior of each.
       - ScoredItem: pair of (ContextItem, float64 score)
    8. Ensure all stub files exist for every SUMMARY.md entry so `mdbook build spec` succeeds.

  <action>
    Extract pipeline stage logic from `CupelPipeline.cs` and write specification chapters. Each chapter follows the dual-audience pattern: conceptual overview → formal definitions → pseudocode → edge cases → conformance notes.

    1. Write `spec/src/pipeline.md`:
       - Overview: the pipeline is a fixed 6-stage transformation from candidate items to ordered context window
       - Mermaid diagram showing data flow between stages (items → classify → scored items → deduplicated → sorted → sliced → placed)
       - State the fixed-order invariant: stages cannot be reordered or skipped
       - State the ordinal-only scoring invariant: scorers rank, slicers drop, placers position

    2. Write `spec/src/pipeline/classify.md` (Stage 1: CLASSIFY):
       - Partition input items into `pinned` and `scoreable` lists
       - Exclude items with `tokens < 0` (exclusion reason: NegativeTokens)
       - Validate pinned items fit within `maxTokens - outputReserve`
       - Pseudocode for CLASSIFY algorithm

    3. Write `spec/src/pipeline/score.md` (Stage 2: SCORE):
       - For each scoreable item, invoke scorer to produce ScoredItem(item, score)
       - Scorer receives the individual item AND the full list of scoreable items (for relative scoring)
       - Score is IEEE 754 float64, conventionally [0.0, 1.0] but not enforced by type
       - Pseudocode for SCORE stage

    4. Write `spec/src/pipeline/deduplicate.md` (Stage 3: DEDUPLICATE):
       - Group by content using byte-exact ordinal comparison (no Unicode normalization, no case folding)
       - For duplicates: keep highest-scoring item; tiebreak by lowest original index
       - Deduplication is enabled by default, can be disabled
       - Pseudocode for DEDUPLICATE algorithm
       - Address pitfall P3 (deduplication identity semantics) explicitly

    5. Write `spec/src/pipeline/sort.md` (Stage 4: SORT):
       - Stable sort descending by score
       - Tiebreak: ascending by original index (preserves input order for equal scores)
       - Implementation pattern: build (score, index) pairs, sort, reconstruct
       - Pseudocode for SORT algorithm
       - Address pitfall P1 (sort stability) explicitly — mandate stable sort or equivalent composite key

    6. Write `spec/src/pipeline/slice.md` (Stage 5: SLICE):
       - Compute effective budget accounting for outputReserve and pinned token cost
       - `effectiveMax = max(0, maxTokens - outputReserve - pinnedTokens)`
       - `effectiveTarget = min(max(0, targetTokens - pinnedTokens), effectiveMax)`
       - Delegate to slicer strategy
       - Link to slicer chapters for strategy-specific algorithms
       - Pseudocode for budget computation and slicer dispatch

    7. Write `spec/src/pipeline/place.md` (Stage 6: PLACE):
       - Merge pinned items (assigned score = 1.0) with sliced items
       - Apply overflow detection based on OverflowStrategy
       - Delegate to placer strategy for final ordering
       - Link to placer chapters for strategy-specific algorithms
       - Pseudocode for merge and overflow handling
  </action>
