  <action>
Update the release workflow with these changes:

1. **Add `id-token: write` permission** at the workflow-level `permissions` block (alongside existing `contents: write`). This is required for the OIDC token exchange with crates.io.

2. **Replace static secret with OIDC auth action** in the publish job. Add a new step BEFORE the "Publish to crates.io" step:
   ```yaml
   - name: Authenticate with crates.io
     uses: rust-lang/crates-io-auth-action@v1
     id: crates-io-auth
   ```
   Then update the "Publish to crates.io" step's env to use the action output:
   ```yaml
   env:
     CARGO_REGISTRY_TOKEN: ${{ steps.crates-io-auth.outputs.token }}
   ```
   Remove the reference to `${{ secrets.CARGO_REGISTRY_TOKEN }}`.

3. **Remove `--locked` flag** from the `cargo publish` command. The Cargo.lock is gitignored (library crate convention per decision 16-01-D3), so `--locked` will error with "the lock file needs to be updated but --locked was passed to prevent this". For library crates, `--locked` is not meaningful — downstream consumers resolve their own dependency versions. The publish command should be:
   ```
   cargo publish --manifest-path crates/cupel/Cargo.toml
   ```

4. **Change `make_latest: false` to `make_latest: true`** in the GitHub Release step. Per the context decision, Rust is the primary distribution going forward.

**Important note on OIDC vs token:** The OIDC auth action will fail on first publish because trusted publishing is not yet configured on crates.io (the crate must exist first). For the FIRST publish only, the user will need to temporarily revert the OIDC step and use `secrets.CARGO_REGISTRY_TOKEN` directly, OR configure `CARGO_REGISTRY_TOKEN` as an environment secret in the `release` environment (which takes precedence). The OIDC action gracefully handles this — when trusted publishing is not configured, it will fail and the workflow will error, prompting the user to use the token approach. After first publish, the user configures trusted publishing on crates.io, and all subsequent publishes use OIDC automatically.

Actually, the cleaner approach: For first publish, the user should add `CARGO_REGISTRY_TOKEN` as a **repository secret** in GitHub. The OIDC action will attempt token exchange, fail (no trusted publisher configured), and the publish step will use the env var from the action output (which will be empty/invalid). This won't work cleanly.

**Revised approach:** Keep the OIDC action but add a fallback. Use this pattern for the publish step:
```yaml
- name: Authenticate with crates.io
  uses: rust-lang/crates-io-auth-action@v1
  id: crates-io-auth
  continue-on-error: true

- name: Publish to crates.io
  run: cargo publish --manifest-path crates/cupel/Cargo.toml
  env:
    CARGO_REGISTRY_TOKEN: ${{ steps.crates-io-auth.outputs.token || secrets.CARGO_REGISTRY_TOKEN }}
```

This way:
- First publish: OIDC fails (continue-on-error), falls back to `secrets.CARGO_REGISTRY_TOKEN`
- After trusted publishing configured: OIDC succeeds, uses OIDC token
- After token deleted: OIDC is the only path (correct final state)
