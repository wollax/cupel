    ```
    ## Overview

    ```

    **Content requirements:**

    1. **Overview** — One-sentence definition: the diagnostics system exposes pipeline selection decisions as structured, inspectable data without affecting output or performance when disabled. Brief paragraph on purpose (explainability, debugging "why was this excluded?").

    2. **Ownership Model** — The trace collector is passed at call time, not stored on the pipeline. One collector instance per pipeline execution. This makes diagnostics per-invocation and avoids shared mutable state.
       - Inline rationale: per-invocation ownership avoids thread-safety concerns and ensures each pipeline run gets a clean diagnostic context.
       - Rejected alternative: storing the collector on the pipeline object — couples lifecycle, prevents concurrent runs with different diagnostic configurations.

    3. **Null-Path Guarantee** — When diagnostics are disabled (NullTraceCollector), no event objects are constructed and no performance overhead is incurred. Implementations achieve this by checking `is_enabled` before constructing event payloads.
       - Inline rationale: diagnostic instrumentation must never regress pipeline performance for callers who don't need it.

    4. **Data Flow** — Mermaid flowchart TD diagram showing:
       - Input candidates → Pipeline (with "trace calls at each stage" annotation) → Output (selected items)
       - Parallel path from Pipeline → TraceCollector → SelectionReport
       - Keep it simple — 5-7 nodes max.

    5. **Summary** — Table listing all diagnostic types and their roles:

       | Type | Role | Defined in |
       |------|------|------------|
       | TraceCollector | Observer contract for pipeline instrumentation | [TraceCollector](diagnostics/trace-collector.md) |
       | TraceEvent | Structured record of a single pipeline observation | [Events](diagnostics/events.md) |
       | PipelineStage | Enumeration of observable pipeline stages | [Events](diagnostics/events.md) |
       | OverflowEvent | Record produced when budget overflow occurs under Proceed strategy | [Events](diagnostics/events.md) |
       | ExclusionReason | Why an item was excluded from the context window | [Exclusion Reasons](diagnostics/exclusion-reasons.md) |
       | InclusionReason | Why an item was included in the context window | [Exclusion Reasons](diagnostics/exclusion-reasons.md) |
       | SelectionReport | Complete diagnostic output from a pipeline run | [SelectionReport](diagnostics/selection-report.md) |

    ---

    **trace-collector.md** — Follow the section pattern from 24-RESEARCH.md:

    ```
    # TraceCollector
    ## Overview
    ## Contract
    ## NullTraceCollector
    ## DiagnosticTraceCollector
    ## TraceDetailLevel
    ## Observer Callback
    ## Conformance Notes
    ```

    Content:

    1. **Overview** — The TraceCollector is an observer interface that pipeline implementations call at instrumentation points. Two built-in implementations: NullTraceCollector (zero-cost disabled path) and DiagnosticTraceCollector (buffered recording with configurable detail level).

    2. **Contract** — Three members, presented as a table:

       | Member | Signature (pseudocode) | Description |
       |--------|----------------------|-------------|
       | `is_enabled` | `-> boolean` | Whether this collector is actively recording. Callers check before constructing event payloads to avoid unnecessary allocations. |
       | `record_stage_event` | `(event: TraceEvent) -> void` | Record a stage-level event. Called once per pipeline stage, after the stage completes. |
       | `record_item_event` | `(event: TraceEvent) -> void` | Record an item-level event. May be filtered by the collector's detail level. |

    3. **NullTraceCollector** — `is_enabled` returns `false`. Both record methods are no-ops. Singleton recommended but not required.
       - Inline rationale: the null collector ensures zero overhead when diagnostics are disabled — no events are allocated, no buffers are grown.

    4. **DiagnosticTraceCollector** — `is_enabled` returns `true`. Accepts a `TraceDetailLevel` parameter (defaults to `Stage`). Buffers all recorded events in insertion order. Exposes accumulated events for report generation after the pipeline run completes.

       Construction parameters (pseudocode table):

       | Parameter | Type | Required | Default | Description |
       |-----------|------|----------|---------|-------------|
       | `detail_level` | TraceDetailLevel | No | `Stage` | Controls which events are recorded |

       Behavior:
       - `record_stage_event`: always records the event.
       - `record_item_event`: records only when `detail_level >= Item`. When `detail_level` is `Stage`, item events are silently discarded.
       - After the pipeline run, the caller extracts a `SelectionReport` from the collector (see [SelectionReport](selection-report.md)).

    5. **TraceDetailLevel** — Enum with ordered values:

       | Value | Ordinal | Description |
       |-------|---------|-------------|
       | `Stage` | 0 | Stage-level events only (durations, item counts) |
       | `Item` | 1 | Stage-level events plus per-item events (individual scores, exclusion reasons) |

       Inline rationale: two levels provide a meaningful performance/detail trade-off. Stage-level is sufficient for performance profiling; Item-level is needed for "why was this excluded?" debugging.
       Rejected alternative: a continuous verbosity integer — discrete levels are more meaningful and prevent ambiguous intermediate values.

    6. **Observer Callback** — Implementations may support an optional observer callback that fires synchronously on each recorded event. The spec defines this as a capability, not a mechanism — implementations choose how to expose the callback (function pointer, closure, delegate, etc.).

       Inline rationale: synchronous firing ensures events are observable in real time for streaming diagnostics. The mechanism is left to implementations because callback abstractions differ fundamentally across languages.

    7. **Conformance Notes:**
       - Implementations MUST provide both a null (disabled) and a diagnostic (enabled) collector.
       - The null collector MUST return `false` from `is_enabled` and MUST NOT allocate or buffer when record methods are called.
       - `record_item_event` MUST be gated by the configured detail level. When `detail_level` is `Stage`, item events MUST be discarded.
       - Implementations MUST preserve event insertion order in the accumulated event list.

    **events.md** — Follow the section pattern:

    ```
    # Events
    ## Overview
    ## TraceEvent
    ## PipelineStage
    ## OverflowEvent
    ## Conformance Notes
    ```

    Content:

    1. **Overview** — Two structural event types: TraceEvent (produced at every instrumentation point) and OverflowEvent (produced only when budget overflow occurs under the Proceed strategy). Each stage fires exactly one stage-level TraceEvent after completing.

    2. **TraceEvent** — Fields table (use context-item.md column format):

       | Field | Type | Required | Default | Description |
       |-------|------|----------|---------|-------------|
       | `stage` | PipelineStage | Yes | — | The pipeline stage that produced this event |
       | `duration_ms` | float64 | Yes | — | Wall-clock duration of the stage in milliseconds. For item-level events, this is `0.0`. |
       | `item_count` | integer | Yes | — | Number of items processed by this stage |
       | `message` | string | No | *(absent)* | Optional diagnostic message. Absent when not provided. |

       Inline rationale for `duration_ms` as float64: milliseconds provide human-readable precision for diagnostic output while remaining portable across all languages. Float64 accommodates sub-millisecond precision without integer overflow concerns.
       Rejected alternative: integer nanoseconds — less readable in diagnostic output; ISO 8601 duration strings — harder to aggregate and compare programmatically.

       **Stage-level event example:**

       ```json
       {
         "stage": "Score",
         "duration_ms": 12.4,
         "item_count": 47
       }
       ```

       Note: `message` is absent because absent fields are omitted (no nulls).

       **Item-level event example:**

       ```json
       {
         "stage": "Slice",
         "duration_ms": 0.0,
         "item_count": 1,
         "message": "BudgetExceeded: 2048 tokens requested, 512 available"
       }
       ```

    3. **PipelineStage** — Enum of observable pipeline stages:

       | Value | Description |
       |-------|-------------|
       | `"Classify"` | Classification and partitioning stage |
       | `"Score"` | Scoring stage |
       | `"Deduplicate"` | Deduplication stage |
       | `"Slice"` | Budget-fitting selection stage |
       | `"Place"` | Final ordering and merge stage |

       Sort is omitted from PipelineStage. Sort is an internal ordering step with no user-visible diagnostic boundary and no meaningful duration to report independently.
       Rejected alternative: including Sort for completeness — adds a stage that no built-in pipeline emits meaningful diagnostics for, increasing conformance burden without diagnostic value.

       Note: implementations may emit events for internal stages not listed here. Callers should handle unknown stage values gracefully.

    4. **OverflowEvent** — Produced when the merged result (pinned + sliced items) exceeds `target_tokens` and `OverflowStrategy` is `Proceed`. See [OverflowStrategy](../data-model/enumerations.md#overflowstrategy).

       Fields table:

       | Field | Type | Required | Default | Description |
       |-------|------|----------|---------|-------------|
       | `tokens_over_budget` | integer | Yes | — | Number of tokens exceeding the target budget |
       | `overflowing_items` | list of ContextItem | Yes | — | All items at the time of overflow (pinned + sliced combined) |
       | `budget` | ContextBudget | Yes | — | The budget that was exceeded |

       JSON example:

       ```json
       {
         "tokens_over_budget": 384,
         "overflowing_items": [
           { "content": "...", "tokens": 512, "kind": "message" },
           { "content": "...", "tokens": 1024, "kind": "tool_output" }
         ],
         "budget": {
           "max_tokens": 4096,
           "target_tokens": 3072
         }
       }
       ```

       OverflowEvent is a data type only. The spec defines its structure but not its delivery mechanism — implementations choose how to surface the event (callback, return value, collector method, etc.).
       Inline rationale: delivery mechanisms differ across languages and concurrency models. The spec prescribes what information is captured, not how it reaches the caller.

    5. **Conformance Notes:**
       - Stage-level `TraceEvent` records MUST have `duration_ms` set to the wall-clock duration of that stage. Item-level events MUST have `duration_ms` set to `0.0`.
       - Each named pipeline stage (Classify, Score, Deduplicate, Slice, Place) MUST produce exactly one stage-level `TraceEvent` per pipeline run, emitted after the stage completes.
       - `OverflowEvent` MUST only be produced when `OverflowStrategy` is `Proceed`. It MUST NOT be produced for `Throw` or any other strategy.
       - Implementations MUST handle unknown `PipelineStage` values gracefully when deserializing diagnostic data from other implementations.

    **Style rules (both files):**
    - No C#, Rust, or language-specific syntax anywhere. Pseudocode only for signatures.
    - MUST keyword only in Conformance Notes sections.
    - snake_case for all JSON field names.
    - Absent fields omitted in JSON examples.
    - Relative links for cross-references (e.g., `[SelectionReport](selection-report.md)`, `[OverflowStrategy](../data-model/enumerations.md#overflowstrategy)`).
