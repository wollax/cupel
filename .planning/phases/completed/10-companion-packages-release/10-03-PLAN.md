  <name>Task 1: Create CI and publish workflows</name>
  <files>
    .github/workflows/ci.yml
    .github/workflows/release.yml
  </files>
  <action>
    **ci.yml** — New PR build+test workflow:
    - Trigger: `push` to main and `pull_request` to main
    - Single job: `build-test`
    - Steps: checkout, setup-dotnet 10.x, `dotnet build --configuration Release`, `dotnet test --configuration Release --no-build`
    - Use `fetch-depth: 0` for MinVer

    **release.yml** — Complete rewrite of existing stub:
    - Trigger: `workflow_dispatch` with optional `dry-run` boolean input (default false)
    - Permissions: `id-token: write` (OIDC), `contents: write` (GitHub Release)

    Job 1: `pack`
    - Checkout with `fetch-depth: 0` (MinVer requires full history)
    - Setup .NET 10.x
    - `dotnet build --configuration Release`
    - `dotnet test --configuration Release --no-build` (run full test suite before packing)
    - `dotnet pack --configuration Release --no-build --output ./nupkg`
    - Upload `./nupkg` as artifact

    Job 2: `consumption-tests` (needs: pack)
    - Download nupkg artifact to `./nupkg`
    - Copy nupkg files to the path that `nuget.config` expects (e.g., `tests/Wollax.Cupel.ConsumptionTests/packages/`)
    - Setup .NET 10.x
    - `dotnet test tests/Wollax.Cupel.ConsumptionTests/ --configuration Release`
    - This job validates the exact packages being published

    Job 3: `publish` (needs: consumption-tests)
    - Condition: `if: inputs.dry-run != true`
    - Download nupkg artifact
    - Setup .NET 10.x
    - NuGet OIDC login via `NuGet/login@v1` (with `user: wollax`)
    - `dotnet nuget push ./nupkg/*.nupkg --api-key ${{ steps.nuget-login.outputs.NUGET_API_KEY }} --source https://api.nuget.org/v3/index.json --skip-duplicate`
    - Get version from nupkg filename or `dotnet msbuild -getProperty:Version`
    - Create GitHub Release via `softprops/action-gh-release@v2` with tag `v$VERSION`
  </action>
  <verify>

    `cat .github/workflows/ci.yml | python3 -c "import sys,yaml; yaml.safe_load(sys.stdin)"` (or equivalent YAML validation).
  </verify>
  <done>
    ci.yml triggers on push/PR, builds and tests. release.yml triggers on manual dispatch, packs, runs consumption tests, and publishes via OIDC.
  </done>

<task type="auto">
  <name>Task 2: Create consumption tests against packed .nupkg files</name>
  <files>
    tests/Wollax.Cupel.ConsumptionTests/Wollax.Cupel.ConsumptionTests.csproj
    tests/Wollax.Cupel.ConsumptionTests/nuget.config
    tests/Wollax.Cupel.ConsumptionTests/ConsumptionTests.cs
  </files>
  <action>
    **IMPORTANT:** The consumption test project must NOT be included in the main solution. It uses PackageReference (not ProjectReference) and resolves from a local .nupkg source. Including it in the solution would cause build failures when packages aren't packed. Either add it to a separate solution or keep it standalone (run via direct `dotnet test` path).

    **nuget.config:**
    ```xml
    <?xml version="1.0" encoding="utf-8"?>
    <configuration>
      <packageSources>
        <clear />
        <add key="local" value="./packages" />
        <add key="nuget.org" value="https://api.nuget.org/v3/index.json" />
      </packageSources>
    </configuration>
    ```
    The `local` source points to a `packages/` directory relative to the test project. In CI, the workflow copies packed .nupkg files here before running tests.

    **ConsumptionTests.csproj:**
    - `OutputType` Exe, `IsPackable` false
    - `ManagePackageVersionsCentrally` false (override — this project uses floating versions)
    - `PackageReference` to `TUnit` with explicit version (not from Directory.Packages.props)
    - `PackageReference` to `Wollax.Cupel` Version="*"
    - `PackageReference` to `Wollax.Cupel.Json` Version="*"
    - `PackageReference` to `Wollax.Cupel.Extensions.DependencyInjection` Version="*"
    - `PackageReference` to `Wollax.Cupel.Tiktoken` Version="*"
    - `PackageReference` to `Microsoft.Extensions.DependencyInjection` with explicit version (for DI container in tests)
    - `PackageReference` to `Microsoft.ML.Tokenizers.Data.O200kBase` with explicit version (for Tiktoken tests)
    - No `ProjectReference` entries — this is the entire point

    **ConsumptionTests.cs** — Tests that prove each package works from NuGet:

    1. **Core pipeline** — Create a `CupelPipeline` from `CupelPresets.Chat()`, execute with minimal items, verify `ContextResult` is returned.

    2. **JSON serialization** — Serialize `CupelPresets.Chat()` via `CupelJsonSerializer.Serialize`, deserialize back, verify scorer count matches.

    3. **DI registration** — `AddCupel` + `AddCupelPipeline`, resolve keyed pipeline, verify not null.

    4. **Tiktoken counting** — `TiktokenTokenCounter.CreateForModel("gpt-4o")`, count tokens for a known string, verify > 0.

    5. **Tiktoken WithTokenCount** — Create a `ContextItem`, call `WithTokenCount`, verify `Tokens` is set.

    These tests are intentionally simple — they prove the packages install, resolve types, and work. Detailed behavior is tested in the unit test projects.

    Do NOT add this project to `Cupel.slnx`. It is a standalone project run only in the publish workflow.
  </action>
  <verify>
    To test locally: `rtk dotnet pack --configuration Release --output tests/Wollax.Cupel.ConsumptionTests/packages/` then `rtk dotnet test tests/Wollax.Cupel.ConsumptionTests/`.
    Consumption tests pass when run against locally packed .nupkg files.
  </verify>
  <done>
    Consumption test project resolves all four packages from local .nupkg source and all tests pass, proving packages are correctly structured and usable.
  </done>
