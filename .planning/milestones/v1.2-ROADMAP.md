# Milestone v1.2: Rust Parity & Quality Hardening

**Status:** 🔄 IN PROGRESS (started 2026-03-15)
**Goal:** Close the diagnostics gap between the Rust and .NET implementations, harden the Rust API surface before semver commits, and batch-address 74+ accumulated quality issues.
**Phases:** 23-32
**Total Plans:** TBD (planned: ~20-26)

## Overview

The Rust `cupel` crate (v1.1.0) is complete at the pipeline level but lacks the diagnostics system that is the primary production-debuggability differentiator of the .NET implementation. v1.2 closes that gap, hardens the API surface for long-term semver stability, and batch-resolves accumulated quality debt across both languages.

Three structural constraints govern sequencing:
1. `#[non_exhaustive]` on `CupelError`/`OverflowStrategy` (Phase 23) must ship before any new error variants are added.
2. The diagnostics spec chapter (Phase 24) must be merged before any diagnostics implementation begins (Phases 25-28).
3. The CI conformance drift guard (Phase 25) must precede diagnostic conformance vectors.

Quality hardening (Phases 30-32) is independent of the diagnostics track and can run in parallel.

---

## Phases

### Phase 23: API Hardening Foundations

**Goal**: Ship the `#[non_exhaustive]` attribute on `CupelError` and `OverflowStrategy`, derive `Debug`/`Clone`/`Copy` on concrete slicer/placer structs, add `ContextKind` factory methods and `TryFrom<&str>`, and expose `ContextBudget::unreserved_capacity()` — all before any new error variants are introduced.
**Depends on**: Phase 22
**Parallel-capable**: Yes (fully independent of spec/diagnostics work)
**Requirements**: RAPI-01, RAPI-02, RAPI-03, RAPI-04, RAPI-05
**Plans**: ~3 plans

**Success Criteria**:
- `match cupel_error { ... }` on `CupelError` or `OverflowStrategy` without a wildcard arm produces a compiler warning (`irrefutable_let_patterns` / exhaustive match) only if the user explicitly opts in — library consumers writing `_` arms compile without warning.
- `GreedySlice`, `KnapsackSlice`, `UShapedPlacer`, and `ChronologicalPlacer` can be copied and printed with `{:?}` without any trait bound imports.
- `ContextKind::message()`, `ContextKind::system_prompt()`, `ContextKind::document()`, `ContextKind::tool_output()`, and `ContextKind::memory()` compile and return the expected variant.
- `ContextKind::try_from("message")` returns `Ok(ContextKind::message())` and `ContextKind::try_from("unknown")` returns `Err(...)`.
- `budget.unreserved_capacity()` returns `max_tokens - output_reserve - sum(reserved_slots)` as `i64`; a budget where reserved slots exceed max tokens returns a negative value without panic.

### Phase 24: Diagnostics Spec Chapter

**Goal**: Author, review, and merge a language-agnostic diagnostics spec chapter in `/spec/` that resolves all load-bearing API decisions before any implementation PR opens.
**Depends on**: Phase 23
**Parallel-capable**: No — blocks Phases 25-29
**Requirements**: SPEC-01
**Plans**: ~2 plans

**Success Criteria**:
- The spec chapter exists at `spec/book/src/diagnostics.md` (or equivalent path) and is included in the mdBook TOC.
- The chapter explicitly documents: ownership model (`&mut dyn TraceCollector` per-call), trait receiver mutability choice, `Send + Sync` requirement decision, `PipelineStage` variant list (with Sort exclusion rationale), `ExclusionReason` variant designs (data-carrying or fieldless with rationale), `SelectionReport` field access pattern, serde wire format for all diagnostic types, module placement (`cupel::diagnostics`), and the `Pipeline::run()` compatibility guarantee.
- A spec reviewer (or author) can answer "what variant does `BudgetExceeded` carry?" by reading the chapter alone, without consulting implementation files.
- The chapter has no `TODO` or `FIXME` markers before merge.

### Phase 25: Conformance Infrastructure & Drift Guard

**Goal**: Establish the CI conformance drift guard that diffs `spec/conformance/` against `crates/cupel/tests/conformance/` and fails on divergence, fix misleading comments in existing conformance vector TOML files, and create the conformance vector schema for diagnostics events.
**Depends on**: Phase 24
**Parallel-capable**: Partial — CONF-01 (comment fixes) can begin before Phase 24; drift guard requires Phase 24 to know diagnostic vector shape
**Requirements**: CONF-01, CONF-02, CI-03, SPEC-02
**Plans**: ~3 plans

**Success Criteria**:
- A CI step (in `ci-rust.yml` or a dedicated `ci-conformance.yml`) runs `diff -rq spec/conformance/required/ crates/cupel/tests/conformance/` (or equivalent) and the job fails if any file differs or is absent from either side.
- The five conformance vector TOML files with misleading/incorrect comments are corrected in both `spec/conformance/` and the vendored copy under `crates/cupel/tests/conformance/`; the drift guard passes after the fix.
- `[expected.diagnostics]` schema is documented in the spec chapter or a companion `conformance/SCHEMA.md`, and at least one example diagnostic conformance vector file exists.
- Manually editing one conformance vector in `spec/conformance/` without updating the vendored copy causes CI to fail within the same PR.

### Phase 26: Diagnostics Data Types

**Goal**: Implement the `cupel::diagnostics` module with all data type definitions — `ExclusionReason`, `InclusionReason`, `PipelineStage`, `TraceEvent`, `IncludedItem`, `ExcludedItem`, and `SelectionReport` — following the spec chapter exactly.
**Depends on**: Phase 25
**Parallel-capable**: No — all subsequent diagnostics phases depend on these types
**Requirements**: DIAG-04, DIAG-05, DIAG-06
**Plans**: ~2 plans

**Success Criteria**:
- `cargo check` and `cargo clippy --all-targets` pass cleanly with the new `diagnostics` module.
- `ExclusionReason` and `TraceEvent` carry `#[non_exhaustive]`; attempting to exhaustively match either enum without a wildcard arm produces a compile error.
- `SelectionReport` fields are accessible only via public accessor methods (not direct field access from outside the crate).
- Unit tests constructing `SelectionReport` from known `IncludedItem` and `ExcludedItem` slices assert correct counts, total token sums, and reason attribution — all passing without a running pipeline.
- `size_of::<ExclusionReason>()` is reported in a test comment (as a baseline for future regression); the struct has no unintended heap allocations in the no-payload variants.

### Phase 27: TraceCollector Trait & Implementations

**Goal**: Implement the `TraceCollector` trait with extended default accumulation methods, `NullTraceCollector` as a zero-sized type, and `DiagnosticTraceCollector` with `Vec<TraceEvent>` buffer and `build_report(self) -> SelectionReport`.
**Depends on**: Phase 26
**Parallel-capable**: No — pipeline integration (Phase 28) requires this trait
**Requirements**: DIAG-01, DIAG-02, DIAG-03
**Plans**: ~2 plans

**Success Criteria**:
- `std::mem::size_of::<NullTraceCollector>() == 0` asserted in a test.
- `NullTraceCollector` implements `TraceCollector` with all no-op methods and zero heap allocation; a monomorphized call to any no-op method inlines to nothing (verified by test existence, not assembly inspection).
- `DiagnosticTraceCollector::build_report(self)` consumes the collector and returns a `SelectionReport` populated from accumulated events; a unit test exercising direct method calls (not via pipeline) asserts the report matches the recorded events.
- The `TraceCollector` trait is object-safe — `&mut dyn TraceCollector` compiles without error.
- `NullTraceCollector` and `DiagnosticTraceCollector` are both re-exported from `cupel::diagnostics`; neither requires an internal module path to use.

### Phase 28: Pipeline Integration & run_traced

**Goal**: Integrate `TraceCollector` into the pipeline by refactoring `place::place_items` overflow handling into the orchestrator, implementing `Pipeline::run_traced()`, and modifying `Pipeline::run()` to delegate to `run_traced(&mut NullTraceCollector)`.
**Depends on**: Phase 27
**Parallel-capable**: No — DryRun (Phase 29) requires this
**Requirements**: DIAG-07, DIAG-08
**Plans**: ~3 plans

**Success Criteria**:
- `Pipeline::run()` signature and behavior are unchanged — all 28 existing conformance tests pass without modification.
- `Pipeline::run_traced(&mut collector)` returns `(Vec<ScoredItem>, SelectionReport)`; calling it with a `NullTraceCollector` produces the same item list as `run()`.
- The `SelectionReport` from `run_traced` with a `DiagnosticTraceCollector` contains one `ExcludedItem` for every item that was deduplicated, budget-excluded, or overflow-truncated, with an accurate `ExclusionReason`.
- `Pipeline::dry_run()` returns only `SelectionReport` without consuming or returning items; calling it with a known input produces the same report as `run_traced` called on the same input (item list ignored).
- Per-stage wall-clock timing appears in `TraceEvent` records; a test asserting `event.duration >= Duration::ZERO` passes for every event in a `DiagnosticTraceCollector` after a pipeline run.

### Phase 29: Diagnostics Polish & Serde Integration

**Goal**: Add serde serialization behind the `serde` feature flag to all diagnostic types, write doc tests for `run_traced` and `dry_run`, confirm `cargo test --all-features` is clean, and write at least one diagnostic conformance vector exercising the `[expected.diagnostics]` schema.
**Depends on**: Phase 28
**Parallel-capable**: No — requires all diagnostic types to be finalized
**Requirements**: DIAG-09
**Plans**: ~2 plans

**Success Criteria**:
- `cargo test --all-features` produces zero failures and zero warnings.
- `serde_json::to_string(&report)` compiles only when the `serde` feature is enabled; without the feature, it is a compile error.
- JSON field names in `SelectionReport` serialization exactly match the names specified in the diagnostics spec chapter.
- A serde roundtrip test (`serialize → deserialize → assert_eq!`) passes for `SelectionReport`, `TraceEvent`, and `ExclusionReason`.
- At least one `[expected.diagnostics]` conformance vector exists in `spec/conformance/` and its vendored copy, and the drift guard passes after adding it.
- The crate-level `lib.rs` re-exports `SelectionReport`, `DiagnosticTraceCollector`, `NullTraceCollector`, and `TraceCollector` from `cupel::diagnostics` without requiring users to navigate sub-modules.

### Phase 30: Quality Hardening — CI

**Goal**: Strengthen the Rust CI pipeline with `clippy --all-targets`, `cargo-deny` unmaintained advisory warnings, and the conformance drift guard (coordinated with Phase 25).
**Depends on**: Phase 23 (for `--all-targets` to pass cleanly)
**Parallel-capable**: Yes — independent of diagnostics track; can run alongside Phases 24-29
**Requirements**: CI-01, CI-02, CI-03 (CI-03 coordinated with Phase 25)
**Plans**: ~2 plans

**Success Criteria**:
- The `ci-rust.yml` workflow runs `cargo clippy --all-targets -- -D warnings`; the job fails if any clippy lint fires on any target (lib, tests, examples, benches).
- `cargo deny check` configuration includes `[advisories] unmaintained = "warn"` (or `"deny"`); a PR introducing a dependency with an unmaintained advisory causes CI to fail or warn as configured.
- CI passes cleanly on the main branch after both changes are applied.
- A test PR introducing a deliberate clippy lint (e.g., `#[allow(dead_code)]` removed from a test-only struct) causes the clippy job to fail — verified by PR description or local `--dry-run` invocation.

**Goal**: Resolve mechanical .NET quality issues: XML doc comment gaps, naming inconsistencies, defensive coding improvements, and the `unreserved_capacity()` parity item for the .NET `ContextBudget`.
**Depends on**: Phase 23 (for RAPI-05 .NET parity)
**Parallel-capable**: Yes — fully independent of Rust diagnostics track
**Requirements**: RAPI-05 (.NET half), RAPI-06 (.NET half), QH-01
**Plans**: ~2 plans

**Success Criteria**:

- `KnapsackSlice` in .NET validates DP table size before allocation and returns `CupelError`/throws with a message identifying the capacity × items cell count when it exceeds 50M.
- All naming inconsistencies flagged in open issues (e.g., mismatched parameter names between interface and implementation) are resolved; `dotnet build --verbosity normal` produces zero warnings.

### Phase 32: Quality Hardening — Rust Codebase

**Goal**: Resolve mechanical Rust quality issues: doc comment gaps, naming consistency, test coverage gaps for edge cases in scorers, `KnapsackSlice` DP table size guard, and `#[must_use]` audit.
**Depends on**: Phase 23 (for `#[non_exhaustive]` prerequisite to `CupelError::TableTooLarge`)
**Parallel-capable**: Yes — fully independent of diagnostics track
**Requirements**: RAPI-06 (Rust half), QH-02, QH-03
**Plans**: ~3 plans

**Success Criteria**:
- `cargo doc --no-deps 2>&1 | grep "^warning"` produces zero output; all public items have doc comments with `# Errors`, `# Panics`, and `# Examples` sections where applicable.
- `KnapsackSlice::slice()` returns `Err(CupelError::TableTooLarge { cells: u64 })` when `capacity × items > 50_000_000`; a unit test with a budget of 50,001 tokens and 1,001 items exercises this path and asserts the error variant.
- Tests for `RecencyScorer` with all-null timestamps, `TagScorer` with an empty tag set, and `FrequencyScorer` with uniform-frequency items all exist and pass.
- All `Result`-returning public functions and builder methods carry `#[must_use]`; `cargo clippy` reports no `unused_must_use` lints.
- `cargo test --all-features` passes with zero failures after all hardening changes.
